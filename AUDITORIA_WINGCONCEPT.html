1. Resumen ejecutivo

WingConcept tiene una base técnica sólida en el backend: autenticación JWT, rate limiting, validación Pydantic, webhooks idempotentes, política de admins bloqueada, headers de seguridad y tests de seguridad automatizados. Sin embargo, el proyecto no está listo para producción comercial al nivel de Apple o Amazon por tres bloques principales:

  • Gestión de secretos: claves live en entorno dev, webhook Stripe vacío, posible misconfiguración Supabase.
  • Despliegue incompleto: falta frontend/Dockerfile, scripts de deploy eliminados, Nginx no enruta al frontend.
  • Modelo de sesión frontend: tokens JWT en localStorage — vulnerable a XSS; no cumple estándar enterprise.

La experiencia visual es atractiva y orientada a marca premium, pero faltan señales de confianza, consistencia i18n, accesibilidad WCAG, estados de error unificados y un checkout con fricción mínima comparable a Amazon.

4. Experiencia de usuario — Benchmark Apple / Amazon

Lo que Apple / Amazon exigen

  • Checkout en ≤3 pasos con progreso claro
  • Confianza: SSL badge, políticas, soporte visible
  • Accesibilidad WCAG 2.1 AA (contraste, teclado, screen readers)
  • Performance: LCP < 2.5s, skeleton loaders, zero layout shift
  • Errores humanos, localizados, con acción de recuperación
  • Búsqueda, filtros, reviews, recomendaciones
  • Guest checkout + cuenta opcional post-compra
  • Emails transaccionales consistentes con marca

Estado actual WingConcept

  • Checkout multi-paso implementado — buena base
  • Diseño premium, animaciones Framer Motion
  • Mezcla EN/ES sin i18n completo
  • Sin guest checkout (login obligatorio)
  • Pocas etiquetas ARIA; nav altura 144px muy grande en mobile
  • Enlaces rotos: "Download Manuals" → #
  • Sin reviews, wishlist, comparador, tracking en tiempo real
  • Admin funcional pero algunas páginas aún stub

6. Fortalezas actuales (mantener)

  • Backend FastAPI bien estructurado: capas service/schema/model, excepciones tipadas, async SQLAlchemy 2.0
  • Seguridad en profundidad: TrustedHost, CORS configurable, CSP, body limit 2MB, X-Session-ID sanitizado
  • Política de administradores restrictiva — alineada con principio de mínimo privilegio
  • Stripe Checkout hosted — PCI scope reducido (no almacenan PAN en servidor)
  • Tests de seguridad: SQL injection, session ID, body size, headers, admin auth
  • Configurador 3D con precios calculados server-side — anti-fraude en productos custom
  • Panel admin amplio: productos, stock, órdenes, usuarios, CMS, cupones, dealers
  • Middleware Next.js para /admin/* con validación server-side de rol
  • Email transaccional vía Resend con dominio verificado

7. Plan de acción priorizado

Fase 0 — Emergencia (24–48 h)

  1. Rotar todas las credenciales expuestas: Stripe (revocar keys live), Supabase DB password, API keys, Resend, JWT SECRET_KEY producción
  2. Separar entornos: sk_test_ en dev, sk_live_ solo en VPS producción
  3. Configurar STRIPE_WEBHOOK_SECRET en Stripe Dashboard → endpoint https://wingconcept.com/api/v1/webhooks/stripe
  4. Verificar Supabase: SERVICE = service_role, ANON = anon (nunca invertidos)
  5. Confirmar backend/.env en .gitignore y que nunca se commiteó (git log -- backend/.env)

Fase 1 — Bloqueantes producción (1 semana)

  1. Crear frontend/Dockerfile (Next.js standalone) y restaurar scripts deploy
  2. Actualizar docker/nginx/nginx.conf con proxy al frontend
  3. Migrar tokens de localStorage a cookies HttpOnly Secure SameSite=Strict
  4. Desplegar en /opt/wingconcept con ENVIRONMENT=production, DEBUG=False, REQUIRE_EMAIL_VERIFIED=true
  5. Activar Sentry + alertas uptime en /health

Fase 2 — Confianza y conversión (2–3 semanas)

  1. Páginas legales: Privacidad, Términos, Envíos, Devoluciones, GDPR/CCPA básico
  2. Guest checkout o registro post-pago (patrón Amazon)
  3. Auditoría Lighthouse + corrección LCP/CLS; reducir nav mobile
  4. i18n completo ES/EN con next-intl o similar
  5. Señales de confianza en checkout: logos pago, soporte tel/email, estimación envío

Fase 3 — Nivel enterprise (1–2 meses)

  1. WCAG 2.1 AA: axe-core en CI, skip links, focus visible, contraste
  2. CAPTCHA adaptativo en login/register tras rate limit
  3. WAF (Cloudflare) delante de VPS — DDoS L7, bot management
  4. Backup automatizado Supabase + runbook incidentes

Veredicto final: WingConcept tiene fundamentos de backend de calidad profesional, pero requiere cerrar brechas de secretos, despliegue Docker completo y hardening de sesiones frontend antes de operar como tienda live con pagos reales. Con las fases 0–1 completadas, el proyecto puede alcanzar un nivel aceptable para lanzamiento controlado (soft launch). Las fases 2–3 acercan la experiencia al estándar Apple/Amazon en confianza, accesibilidad y conversión.
